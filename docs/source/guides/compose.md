# Compose (Process-Bigraph) Simulations

*Available since v0.9.0*

The **compose** subsystem enables running reproducible biological simulations
using the [process-bigraph](https://github.com/vivarium-collective/process-bigraph)
framework. It supports multiple simulation engines --- including **v2ecoli**
(whole-cell *E. coli*), **COPASI**, **Tellurium**, and **ReaDDy** --- through a
unified API and CLI.

## Overview

Compose simulations use a different architecture from the vEcoli batch workflow:

| Aspect | vEcoli (Batch) | Compose |
|--------|----------------|---------|
| **Input** | Config JSON + git repo | OMEX/PBG/SBML file upload or curated templates |
| **Engine** | Nextflow + vEcoli | process-bigraph + Singularity |
| **Container** | Pre-built from git hash | Auto-generated from PBG dependencies |
| **Endpoints** | `/api/v1/simulations` | `/compose/v1/simulation/*` |
| **CLI** | `atlantis simulation` | `atlantis compose` |

### How it works

1. **You submit** a simulation request (OMEX archive, PBG file, SBML file, or a
   curated template like v2ecoli)
2. **The API** auto-generates a Singularity container definition based on the
   simulation's Python dependencies
3. **If the container doesn't exist yet**, a SLURM job builds it on the HPC
   cluster (~10--15 minutes for first build; cached thereafter)
4. **The simulation runs** inside the container on the HPC via SLURM
5. **Results** are zipped and available for download

## Quick Start: v2ecoli Whole-Cell E. coli

The v2ecoli curated simulator is the flagship compose workflow. It runs a
whole-cell *E. coli* simulation composed of 55 biological processes wired
together via process-bigraph.

### Submit and poll

```bash
uv run atlantis compose ecoli \
    --duration 60 \
    --seed 0 \
    --poll \
    --base-url https://sms.cam.uchc.edu
```

This command:
- Submits a 60-second whole-cell simulation with random seed 0
- Builds the Singularity container if this is the first run (~15 min)
- Polls every 10 seconds until the job completes
- Prints the simulation ID for result retrieval

### Check status

```bash
uv run atlantis compose status <SIMULATION_ID> \
    --base-url https://sms.cam.uchc.edu
```

### Download results

```bash
uv run atlantis compose results <SIMULATION_ID> \
    --dest ./compose_output \
    --base-url https://sms.cam.uchc.edu
```

Results are downloaded as a ZIP archive containing the simulation output
(e.g., `final_state.json` for v2ecoli).

### Retrieve the process-bigraph document

Every simulation's input document is persisted. You can retrieve it later:

```bash
uv run atlantis compose doc <SIMULATION_ID> \
    --base-url https://sms.cam.uchc.edu
```

### v2ecoli options

| Option | Default | Description |
|--------|---------|-------------|
| `--duration` | `60.0` | Simulation duration in seconds |
| `--seed` | `0` | Random seed for stochastic processes |
| `--interval` | `1.0` | Execution timestep in seconds |
| `--features` | `[]` | JSON list of feature modules (e.g., `'["ppgpp_regulation"]'`) |
| `--cache-dir` | `/out/cache` | Absolute path to pre-computed ParCa cache inside the container |
| `--poll / --no-poll` | `--no-poll` | Wait for completion |

### What happens under the hood

When you run `atlantis compose ecoli`:

1. The API generates an OMEX archive containing a `v2ecoli.pbg` file ---
   a process-bigraph document that configures the simulation parameters
   (seed, features, cache path)

2. A Singularity container definition is auto-generated by
   [pbest](https://github.com/biosimulations/pbest) with extra pip
   dependencies injected:
   - The base container includes COPASI, Tellurium, ReaDDy, and pbest
   - For v2ecoli, `git+https://github.com/vivarium-collective/v2ecoli.git`
     is appended (along with its transitive dependency vEcoli)

3. The container definition is hashed. If a container with that hash
   already exists on the HPC, the build is skipped. Otherwise, a SLURM
   job builds the Singularity image (~15 minutes)

4. A Python runner script is generated and uploaded to the experiment
   directory on the HPC:
   ```python
   from v2ecoli.composite import make_composite
   composite = make_composite(cache_dir='/out/cache', seed=0, features=[])
   composite.run(60.0)
   ```

5. A SLURM batch job executes the runner inside the container via
   `singularity exec`, with bind mounts for:
   - The experiment directory (`/experiment`)
   - The pre-computed ParCa cache (`/out/cache`)

6. The simulation output (`final_state.json`) is zipped into `results.zip`
   for download

### ParCa cache

The v2ecoli model requires a pre-computed **ParCa (Parameter Calculator)
cache** containing:

- `initial_state.json` --- baseline cell state (molecule counts, parameters)
- `sim_data_cache.dill` --- serialized configs for all 55 biological processes
- `cache_version.json` --- hash of source files for staleness detection

The cache is hosted on the HPC filesystem and bind-mounted into the container
at `/out/cache`. It is generated from the
[v2ecoli fixture](https://github.com/vivarium-collective/v2ecoli/blob/main/models/parca/parca_state.pkl.gz)
using the same v2ecoli version installed in the container (to avoid
`StaleCacheError` hash mismatches).

```{note}
The ParCa cache must be rebuilt whenever the v2ecoli package is updated.
The cache version check (`verify_cache_version`) catches mismatches and
gives a clear error message.
```

## Generic Simulation (Custom PBG/OMEX)

Upload your own OMEX archive, PBG file, or SBML file:

```bash
uv run atlantis compose run mymodel.omex \
    --interval-time 1.0 \
    --poll \
    --base-url https://sms.cam.uchc.edu
```

| Option | Default | Description |
|--------|---------|-------------|
| `--interval-time` | `1.0` | Simulation interval/duration |
| `--batch / --no-batch` | `--no-batch` | Batch submission mode |
| `--poll / --no-poll` | `--no-poll` | Wait for completion |

### OMEX archives

An OMEX file is a ZIP archive that bundles:
- A `.pbg` file (process-bigraph document)
- Optional `.sbml` files (model definitions)

The API extracts the `.pbg` file to determine container dependencies and
persists the document for later retrieval.

## Process-Bigraph Document Format

Compose simulations use `.pbg` files --- JSON documents describing composable
simulation models:

```json
{
    "state": {
        "my_process": {
            "_type": "process",
            "address": "local:module.ClassName",
            "config": { "param": 42 },
            "interval": 1.0,
            "inputs": { "substrate": ["stores", "substrate"] },
            "outputs": { "product": ["stores", "product"] }
        }
    }
}
```

Key concepts:

- **Processes** --- continuous compute units (ODE solvers, etc.)
- **Steps** --- discrete one-shot compute units
- **Ports** --- named I/O connections (`inputs`/`outputs`)
- **Stores** --- shared state tree; ports wire to store paths
- **Address** --- resolution path for the compute class

`.pbg` files can be submitted standalone or bundled inside an OMEX archive.

## Architecture

### Container auto-generation

The compose subsystem uses [pbest](https://github.com/biosimulations/pbest)
to automatically generate Singularity container definitions from a PBG
document's dependencies. The generated definition:

1. Starts from a `uv` + Python 3.12 base image
2. Installs pbest and its runtime
3. Installs simulation-specific dependencies via micromamba/pip
4. For v2ecoli: additionally installs `v2ecoli` from GitHub

Container definitions are content-hashed. If a container with the same hash
already exists on the HPC, the build is skipped entirely.

### Database

Compose uses the same PostgreSQL instance as the vEcoli batch subsystem,
with separate `compose_`-prefixed tables:

- `compose_simulators` --- container definition registry
- `compose_simulations` --- simulation metadata and input documents
- `compose_hpcruns` --- SLURM job tracking
- `compose_packages` --- registered process-bigraph packages
- `compose_bigraph_compute` --- registered processes and steps
- `compose_worker_events` --- real-time event stream (optional NATS)

### Execution flow

```text
Client request
    |
    v
FastAPI endpoint (/compose/v1/curated/ecoli)
    |
    v
generate_container_def_file (pbest)
    |--- inject extra pip deps (v2ecoli)
    |--- hash the definition
    v
Check DB: does this container exist?
    |
    |-- NO --> SLURM: build Singularity image
    |          wait for completion
    |
    |-- YES --> skip build
    v
Upload experiment files to HPC
    |--- OMEX input file
    |--- v2ecoli_run.py (for v2ecoli mode)
    v
SLURM: submit simulation job
    |--- singularity exec --compat ...
    |--- bind mounts: /experiment, /out/cache
    v
Simulation runs inside container
    |--- make_composite(cache_dir, seed, features)
    |--- composite.run(duration)
    v
Output saved to /experiment/output/
    |--- zip to results.zip
    v
Client downloads results.zip
```

## Notes

- Compose endpoints are independent of the existing vEcoli simulation endpoints.
  Both can run simultaneously.
- Compose uses the same PostgreSQL database (separate `compose_`-prefixed tables).
- Container builds and simulation jobs run on HPC via SLURM.
- Optional NATS messaging for real-time worker event streaming.
- The first simulation with a new container definition triggers a build
  (~15 minutes). Subsequent simulations reuse the cached container.
