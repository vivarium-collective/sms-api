Contribution 1 — Stateful Process Runtime NEW • v0.9.3

Mirrors the ../rest-process server paradigm inside sms-api's /compose/v1/ router. Every process and step registered in allocate_core()'s link registry is addressable by name. Instances are created on-demand, driven step-by-step, and terminated explicitly — without taking rest-process as a dependency.

Core singleton + instance store in sms_api/compose/process_runtime.py. Exact mirror of rest-process/rest_process/server.py patterns. 

# sms_api/compose/process_runtime.py
from process_bigraph import allocate_core
from bigraph_schema import Edge
import uuid

# Lazy singleton — same pattern as rest-process/start.py
_core = None
def get_core():
    global _core
    if _core is None:
        _core = allocate_core()  # discovers 75 processes + 42 types
    return _core

# UUID-keyed instance store (mirrors rest-process `processes = {}`)
_instances: dict[str, Any] = {}

def initialize_process(process_name: str, config: dict) -> str:
    core = get_core()
    process_class = core.link_registry.get(process_name)  # same as rest-process
    if process_class is None:
        raise KeyError(f"Process '{process_name}' not found")
    process_id = str(uuid.uuid4())
    instance = process_class(config, core=core)  # standard PBG instantiation
    _instances[process_id] = instance
    return process_id

def update_process(process_id: str, state: dict, interval: float):
    instance = _instances[process_id]
    return instance.update(state, interval)  # standard PBG update
#!/usr/bin/env python3
# Full lifecycle using the Grow process (no SBML needed)
import httpx

BASE = "https://sms.cam.uchc.edu"

# 1. Inspect config schema
r = httpx.get(f"{BASE}/compose/v1/process/Grow/config-schema")
print(r.json())  # {"rate": "float"}

# 2. Instantiate
r = httpx.post(f"{BASE}/compose/v1/process/Grow/initialize",
               json={"config": {"rate": 0.1}})
pid = r.json()["process_id"]

# 3. Inspect I/O schema
print(httpx.get(f"{BASE}/compose/v1/process/Grow/inputs/{pid}").json())
# → {"mass": "float"}
print(httpx.get(f"{BASE}/compose/v1/process/Grow/outputs/{pid}").json())
# → {"mass_delta": "float"}

# 4. Drive with updates
for t in range(5):
    r = httpx.post(f"{BASE}/compose/v1/process/Grow/update/{pid}",
                   json={"state": {"mass": 1.0 + t * 0.1}, "interval": 1.0})
    print(f"t={t}: mass_delta={r.json()['mass_delta']:.4f}")

# 5. Terminate
httpx.post(f"{BASE}/compose/v1/process/Grow/end/{pid}")

Known Gaps & Roadmap Honest

This section exists because the audience will ask. These are not surprises — they are expected next steps.

Simulator steps require SBML in REST update loop

CopasiUTCStep and TelluriumUTCStep require model_source (a valid SBML file path) at instantiation. The init → update → end REST lifecycle works for lightweight processes (Grow, GillespieEvent, MSEComparison). For BioSim simulators, the correct path is biomodels-run / curated/copasi, which handle SBML fetch + SLURM dispatch.

Planned: POST /compose/v1/process/{name}/initialize-with-file accepting multipart SBML upload + config JSON.

Process instances are ephemeral (in-memory)

Instances live in the API pod's Python process. Pod restart or new replica = lost instances. Callers receive 404 on stale UUIDs.

Planned: Redis-backed instance config store using the existing MessagingService in sms_api/dependencies.py. Process re-instantiation on miss (GET /inputs/{id} triggers reload from Redis).

BioModels SLURM job — completion not yet confirmed
